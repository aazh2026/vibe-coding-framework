# Vibe Coding 框架

基于"道法术器"四层结构的 AI 辅助编程方法论。

## 核心框架

```
┌─────────────────────────────────────────┐
│  道 - 方向：做什么、为什么做、目标是什么      │
├─────────────────────────────────────────┤
│  法 - 架构：模块划分、接口设计、数据流        │
├─────────────────────────────────────────┤
│  术 - 调试：问题定位、修复验证、回归测试       │
├─────────────────────────────────────────┤
│  器 - 工具：IDE、AI、辅助工具的最优配置       │
└─────────────────────────────────────────┘
```

---

## 道：方向层

**原则：** 先想清楚要做什么，确保每一步都有明确方向。

### 启动前必须明确

```markdown
## 项目定义

### 目标
[一句话描述最终要交付什么]

### 用户场景
- 谁会用？
- 解决什么问题？
- 在什么环境下使用？

### 边界
- 包含什么：[功能范围]
- 不包含什么：[明确排除的功能]

### 成功标准
- [ ] 功能完成标准
- [ ] 性能要求
- [ ] 验收测试通过
```

### 任务分解

将大目标拆分为可验证的小任务：

```
目标：做一个任务管理应用
    ├── 任务1：设计数据模型（可验证：模型定义 + 测试）
    ├── 任务2：实现 CRUD API（可验证：API 测试通过）
    ├── 任务3：前端界面（可验证：界面截图 + 交互测试）
    └── 任务4：端到端集成（可验证：完整流程演示）
```

### 反模式

- ❌ "做一个类似 XXX 的产品"
- ❌ 边做边想需求
- ❌ 目标频繁变更

## 法：架构层

**原则：** 模块独立，接口先行，整体结构清晰。

### 架构设计检查清单

```markdown
## 架构设计

### 模块划分
- [ ] 每个模块有单一职责
- [ ] 模块间依赖关系清晰（画依赖图）
- [ ] 避免循环依赖

### 接口设计
- [ ] 接口先于实现定义
- [ ] 输入输出类型明确
- [ ] 错误处理策略一致

### 数据流
- [ ] 数据流向清晰（单向还是双向）
- [ ] 状态管理策略确定
- [ ] 持久化方案选定

### 技术选型
- [ ] 核心框架/库选定及理由
- [ ] 数据库选型及理由
- [ ] 部署方案确定
```

### 模块化原则

```typescript
// 好的模块：高内聚、低耦合
// user-service.ts
export interface UserService {
  createUser(input: CreateUserInput): Promise<User>;
  getUser(id: string): Promise<User | null>;
  updateUser(id: string, input: UpdateUserInput): Promise<User>;
}

// 实现隐藏细节，暴露清晰接口
export class UserServiceImpl implements UserService {
  constructor(private db: Database, private logger: Logger) {}
  // ... 实现
}
```

### 接口先行示例

```markdown
## API 接口定义（先写这个）

### POST /api/tasks
- Request: { title: string, description?: string, dueDate?: Date }
- Response: { id: string, title: string, ... }
- Errors: 400(验证失败), 401(未授权)

### GET /api/tasks/:id
- Response: Task | 404
```

## 术：调试层

**原则：** 预期结果 vs 实际结果对比，AI 定向修复。

### 调试流程

```
问题出现
    ↓
① 收集证据
   - 错误信息
   - 日志输出
   - 环境状态
   - 复现步骤
    ↓
② 形成假设（至少3个）
   - 假设A：[可能原因]
   - 假设B：[可能原因]
   - 假设C：[可能原因]
    ↓
③ 验证假设
   - 设计最小复现
   - 逐个排除
    ↓
④ 实施修复
    ↓
⑤ 验证解决 + 回归测试
```

### 调试模板

```markdown
## Bug 报告

### 现象
[实际看到什么]

### 预期
[应该看到什么]

### 复现步骤
1. [步骤1]
2. [步骤2]
3. [步骤3]

### 证据
```
[错误日志]
```

### 环境
- 版本：[commit/版本号]
- 配置：[相关配置]

### 已尝试
- [尝试1] - 结果：
- [尝试2] - 结果：

### 当前假设
[最可能的原因]
```

### AI 辅助调试技巧

1. **给 AI 完整上下文**
   ```
   错误：[完整错误信息]
   代码：[相关代码片段]
   已尝试：[你已做的尝试]
   问题：[具体问 AI 什么]
   ```

2. **对比分析**
   ```
   预期输出：[期望的结果]
   实际输出：[实际的结果]
   差异：[AI 分析差异]
   ```

3. **逐步验证**
   - 先验证输入是否正确
   - 再验证中间处理
   - 最后验证输出

## 器：工具层

**原则：** IDE、AI 模型、辅助工具的最优配置。

### 推荐工具栈

| 类别 | 工具 | 用途 |
|------|------|------|
| **IDE** | Cursor / Claude Code / VS Code + Copilot | 代码编写 |
| **AI** | Claude 3.5 Sonnet / GPT-4 / o1 | 代码生成 |
| **终端** | Warp / iTerm2 / Terminal | 命令执行 |
| **API 测试** | Postman / HTTPie / curl | 接口调试 |
| **数据库** | TablePlus / DataGrip / psql | 数据查看 |
| **Git** | GitHub Desktop / lazygit / CLI | 版本控制 |
| **文档** | Notion / Obsidian / Markdown | 知识管理 |

### 工具配置原则

1. **AI 上下文固定**
   ```markdown
   # .cursorrules / CLAUDE.md
   
   ## 项目上下文
   - 技术栈：[React + TypeScript + Node]
   - 架构：[前端/后端分离，REST API]
   - 代码风格：[具体规则]
   - 常见模式：[项目中反复出现的模式]
   ```

2. **快捷键优化**
   - AI 补全：`Tab` / `Ctrl+K`
   - AI 聊天：`Cmd+L` / `Cmd+I`
   - 快速跳转：`Cmd+P` / `Cmd+Shift+F`

3. **自动化脚本**
   ```json
   {
     "scripts": {
       "dev": "next dev",
       "test": "vitest",
       "lint": "eslint . --fix",
       "typecheck": "tsc --noEmit"
     }
   }
   ```

## 递归优化机制

```
生成器（产出初稿）
    ↓
优化器（代码审查、性能优化、边界处理）
    ↓
验证器（测试通过、类型检查、lint）
    ↓
合入代码库
```

### 角色定义

| 角色 | 职责 | 提示词要点 |
|------|------|-----------|
| **生成器** | 快速实现功能 | "先实现基本功能，不追求完美" |
| **优化器** | 代码审查、优化 | "检查代码质量、性能、边界情况" |
| **验证器** | 测试验证 | "确保测试通过，无类型错误" |

### 优化检查清单

- [ ] 代码可读性
- [ ] 错误处理完备
- [ ] 边界情况处理
- [ ] 性能无明显问题
- [ ] 测试覆盖
- [ ] 类型安全
- [ ] 命名规范

## 实操模板

### 新项目启动

```markdown
# 道：项目定义
- 目标：做一个任务管理应用
- 用户：个人开发者
- 边界：只做核心 CRUD，不做协作功能

# 法：架构设计
- 前端：Next.js + Tailwind
- 后端：Next.js API Routes
- 数据库：Prisma + SQLite
- 模块：tasks/, users/, auth/

# 术：调试策略
- 日志：console.log + 浏览器 DevTools
- 测试：Vitest + React Testing Library
- 断点：VS Code debugger

# 器：工具配置
- IDE：Cursor
- AI：Claude 3.5 Sonnet
- 终端：Warp
- Git：lazygit
```

### 日常开发流程

1. **道**：明确今天要完成什么任务
2. **法**：检查是否影响现有架构
3. **术**：编码 → 调试 → 验证
4. **器**：用 AI 加速，用工具验证

---

## 总结

**道法术器**不是严格的先后顺序，而是四个维度同时考虑：

- 做之前想清**道**（方向）
- 做之时遵循**法**（架构）
- 遇到问题用**术**（调试）
- 全程善用**器**（工具）

目标：把 AI 从随意生成代码，变成稳定、可用、越用越顺手的开发伙伴。
